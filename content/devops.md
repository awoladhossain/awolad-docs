# 🚀 DevOps & Cloud-Native Infrastructure

আধুনিক এন্টারপ্রাইজ সিস্টেমে ডেভঅপ্স (DevOps) এবং ইনফ্রাস্ট্রাকচার কেবল কোড ডিপ্লয়মেন্টের মাধ্যম নয়; এটি হলো একটি সিস্টেমের স্থায়িত্ব (Reliability), কর্মক্ষমতা (Performance) এবং স্কেলিং বাউন্ডারি নিশ্চিত করার হার্টবিট। ডিস্ক রাইট অপারেশন, নেটওয়ার্ক রিড ব্লকিং, কনকারেন্সি লক এবং হার্ডওয়্যার মেমোরির সাথে ক্লাউড কন্টেইনার কীভাবে লড়াই করে, তার পূর্ণাঙ্গ এবং গভীর বিবরণ নিয়ে তৈরি এই হ্যান্ডবুক।

## Module 1: Infrastructure as Code & System Provisioning (ইনফ্রাস্ট্রাকচার এজ কোড ও প্রভিশনিং)

ইনফ্রাস্ট্রাকচারকে ডিক্লেয়ারেটিভ কোডে রূপান্তর, গিটঅ্যপ্স কন্টিনুয়াস ডেলিভারি এবং সার্ভারলেস ইঞ্জিনের কার্নেল-লেভেল আইসোলেশন ও এজ কম্পিউটিং নিয়ে আমাদের এই মডিউলের মূল আলোচনা।

## ১. Declarative vs Imperative IaC ও Terraform Engine Internals

ইনফ্রাস্ট্রাকচার এজ কোড (IaC) হলো কোডের মাধ্যমে ক্লাউড ও অন-প্রেমিস রিসোর্স (যেমন: VM, Network, Database) প্রভিশন এবং ম্যানেজ করার বৈপ্লবিক ধারণা।

### ক. Declarative (ডিক্লেয়ারেটিভ) বনাম Imperative (ইম্পারেটিভ) IaC
১. **Imperative IaC (উদা: Ansible, Chef, Bash):**
   * এটিতে রিসোর্স কীভাবে তৈরি করতে হবে তার নির্দিষ্ট পদক্ষেপ বা কমান্ডের ক্রম (Sequence of instructions) বলে দিতে হয় (উদা: "প্রথমে একটি ভিএম তৈরি করো, তারপর জাভা ইন্সটল করো, তারপর পোর্ট ৮০ ওপেন করো")।
   * **অসুবিধা:** স্টেট ম্যানেজমেন্ট জটিল। অ্যাপের বর্তমান অবস্থা কী তা ট্র্যাক করা কঠিন এবং একই স্ক্রিপ্ট দুইবার রান করলে কনফিগ ডুপ্লিকেট বা ক্র্যাশ হতে পারে (Not idempotent)।
২. **Declarative IaC (উদা: Terraform, CloudFormation):**
   * এটিতে সিস্টেমের কাঙ্ক্ষিত শেষ অবস্থা (Desired End State) বা ব্লুপ্রিন্ট ডিফাইন করা হয় (উদা: "আমার একটি ৩-নোডের কুবেরনেটিস ক্লাস্টার এবং ১টি PostgreSQL ডাটাবেস প্রয়োজন")।
   * **সুবিধা:** সিস্টেমের বর্তমান অবস্থা যাই হোক না কেন, IaC ইঞ্জিন রানটাইমে হিসাব করে কাঙ্ক্ষিত অবস্থায় নিয়ে যায়। এটি পুরোপুরি আইডেমপোটেন্ট (Idempotent)।

### খ. Terraform Core Engine Internals: DAG এবং State Management

#### ১. Directed Acyclic Graph (DAG) ইঞ্জিন:
টেরাফর্ম যখন তার কনফিগারেশন ফাইলগুলো (`.tf`) রিড করে, সে কোডের ক্রমানুসারে কাজ করে না। বরং সে সমস্ত রিসোর্সের একে অপরের ওপর নির্ভরতা (Dependencies) এনালাইসিস করে মেমরিতে একটি **Directed Acyclic Graph (DAG)** বা নির্দেশিত চক্রহীন গ্রাফ তৈরি করে।

```
      [VPC Config]
      /          \
     v            v
[Subnet A]    [Subnet B]
     \            /
      v          v
   [Kubernetes Cluster]
            |
            v
   [Deploy Node Group]
```

* **প্যারালাল এক্সিকিউশন:**
  DAG থেকে টেরাফর্ম বুঝতে পারে যে Subnet A এবং Subnet B এর মধ্যে কোনো সরাসরি নির্ভরতা নেই, তাই সে মেমরিতে একাধিক থ্রেড ওপেন করে সমান্তরালে (Parring/Parallel execution) সাবনেট দুটি তৈরি করে ফেলে। এটি ক্লাউড প্রভিশনিং টাইম প্রায় ৮০% কমিয়ে দেয়।

#### ②. Terraform State Management ও State Locking:
টেরাফর্ম ক্লাউডের রিয়েল রিসোর্সের সাথে কোডের ম্যাপিং বজায় রাখতে একটি JSON ফাইল তৈরি করে, যাকে **State File (`terraform.tfstate`)** বলে। এটি সিস্টেমের একক সত্যের উৎস (Single Source of Truth)।

* **State Locking (স্টেট লকিং):**
  টিমের একাধিক প্রকৌশলী যখন একসাথে একই ইনফ্রাস্ট্রাকচারে কাজ করেন, তখন একই সাথে দুজনে `terraform apply` রান করলে স্টেট ফাইল ক্র্যাশ বা ওভাররাইট হতে পারে।
  * **সমাধান:** **Distributed State Locking**। এন্টারপ্রাইজ সিস্টেমে স্টেট ফাইল S3 বা GCS এর মতো রিমোট স্টোরেজে রাখা হয় এবং সাথে **AWS DynamoDB** বা Redis ব্যবহার করা হয় লক ডিরেক্টরি হিসেবে। কেউ যখন কমান্ড রান করে, টেরাফর্ম DynamoDB-তে একটি ইউনিক লক আইডি (Lock ID) রাইট করে লক একোয়ার করে। কাজ শেষ না হওয়া পর্যন্ত অন্য কারও কমান্ড রানটাইম ব্লক থাকে।
* **State Drift (স্টেট ড্রিফট):**
  কেউ যদি টেরাফর্মের বাইরে ক্লাউড কনসোলে গিয়ে ম্যানুয়ালি কোনো সিকিউরিটি গ্রুপের পোর্ট চেঞ্জ করে ফেলে, টেরাফর্ম তার পরবর্তী রান-এ `terraform plan` এর সময় স্টেট ফাইলের সাথে ক্লাউডের রিয়েল এপিআই রেসপন্স তুলনা করে সেই পরিবর্তনকে **Drift** হিসেবে চিহ্নিত করে এবং কোড অনুযায়ী ক্লাউডকে পুনরায় আগের অবস্থায় নিয়ে যায়।

### গ. বাস্তব ডিক্লেয়ারেটিভ Terraform কনফিগ ডেমো
নিচে একটি অত্যন্ত সুরক্ষিত VPC ও সাবনেট ডিক্লেয়ার করার স্ট্যান্ডার্ড টেরাফর্ম কোড দেওয়া হলো:

```hcl
# AWS Provider Configuration
provider "aws" {
  region = "us-east-1"
}

# ১. VPC ডিক্লেয়ারেশন (Desired State)
resource "aws_vpc" "main_vpc" {
  cidr_block           = "10.0.0.0/16"
  enable_dns_hostnames = true

  tags = {
    Name = "Enterprise-Production-VPC"
  }
}

# ২. পাবলিক সাবনেট ডিক্লেয়ারেশন (VPC এর ওপর নির্ভরশীল)
resource "aws_subnet" "public_subnet" {
  vpc_id                  = aws_vpc.main_vpc.id # এখানে ডাইনামিক ডিপেন্ডেন্সি তৈরি হচ্ছে
  cidr_block              = "10.0.1.0/24"
  map_public_ip_on_launch = true

  tags = {
    Name = "Public-Subnet-A"
  }
}

# ৩. আউটপুট এক্সপোজ করা
output "vpc_id" {
  value       = aws_vpc.main_vpc.id
  description = "The ID of our newly provisioned VPC"
}
```

## ২. GitOps ও Continuous Delivery (ArgoCD & Canary Deployments)

আধুনিক ক্লাউড-নেটিভ ইনফ্রাস্ট্রাকচারে ম্যানুয়াল বা স্ক্রিপ্টেড সিআই/সিডি পাইপলাইনের অবসান ঘটিয়ে **GitOps** ডিজাইন প্যাটার্ন জায়গা করে নিয়েছে।

### ক. Push-based বনাম Pull-based Continuous Delivery (CD)
১. **Push-based CD (উদা: Traditional Jenkins, GitHub Actions):**
   * এখানে বিল্ড পাইপলাইন রান হওয়ার পর রানার নিজেই SSH বা Kubeconfig ক্রেডেনশিয়াল ব্যবহার করে টার্গেট ক্লাস্টারে কানেক্ট হয় এবং নতুন ইমেজ পুশ বা ডিপ্লয় করে।
   * **সিকিউরিটি বিপর্যয়:** আপনার ক্লাস্টারের পূর্ণ অ্যাডমিন এক্সেস কী (Kubeconfig) এক্সটারনাল সিআই/সিডি টুল বা গিটহাব রানারে স্টোর করে রাখতে হয়। রানার হ্যাক হলে সম্পূর্ণ ইনফ্রাস্ট্রাকচার হ্যাক হতে পারে।
২. **Pull-based CD (উদা: ArgoCD, Flux) - GitOps মডেল:**
   * এখানে ক্লাস্টারের ভেতরে একটি ইন্টারনাল কন্ট্রোলার বা এজেন্ট রান করে। এই কন্ট্রোলার বাইরের গিট রিপোজিটরির কনফিগারেশন ট্র্যাক করে।
   * **সুবিধা:** ক্লাস্টারের বাইরে কোনো ক্রেডেনশিয়াল পাস করতে হয় না। ইন্টারনাল কন্ট্রোলার নিজেই গিট থেকে ডেটা পুল করে ক্লাস্টারকে সিঙ্ক করে।

### খ. ArgoCD Reconciliation Loop ও State Engine
ArgoCD একটি কুবেরনেটিস কাস্টম কন্ট্রোলার হিসেবে কাজ করে। এর মূল হার্টবিট হলো **Reconciliation Loop (রিঅ্যাকশন লুপ)**:

```
[Git Config (Desired State)] <--- Compare ---> [Kubernetes Cluster (Actual State)]
                                    |
                            Reconcile Loop (3m)
                                    |
                    Out-of-Sync? -> Auto-Sync Apply
```

১. **Desired State (গিট রিপোজিটরি):** গিটে রাখা কুবেরনেটিস ডিক্লেয়ারেটিভ ইয়ার্মেল (`YAML`) ফাইল।
۲. **Actual State (লাইভ ক্লাস্টার):** কুবারনেটিস ক্লাস্টারে বর্তমানে রিয়েল-টাইমে রানিং পড ও সার্ভিস।
৩. ArgoCD কন্ট্রোলার প্রতি ৩ মিনিটে (বা ওয়েবহুকের মাধ্যমে ইনস্ট্যান্ট) লাইভ ক্লাস্টারের মেটাডাটা ও গিটের স্টেট মেমরিতে তুলনা করে।
৪. কোনো অমিল খুঁজে পেলে সে ক্লাস্টারকে **`OutOfSync`** দেখায় এবং অটো-সিঙ্ক অন থাকলে গিট অনুযায়ী কুবেরনেটিস এপিআই হিট করে ক্লাস্টারকে রি-অ্যালাইন করে নেয়। কেউ ম্যানুয়ালি ক্লাস্টারে কমান্ড দিয়ে কন্টেইনার ডিলিট করলেও ArgoCD তা সেকেন্ডের মধ্যে গিট রিড করে পুনরায় পড তৈরি করে ফেলে!

### গ. Canary & Blue-Green Deployment Strategies

#### ১. Blue-Green Strategy:
মেমরিতে দুটি হুবহু পরিবেশ থাকে—**Blue (রানিং প্রোডাকশন ট্রাফিক)** এবং **Green (নতুন ভার্সন)**। নতুন ভার্সন গ্রিন-এ ডিপ্লয় করার পর সমস্ত টেস্ট পাস হলে লোড ব্যালেন্সারের রাউটিং ১ সেকেন্ডে ব্লু থেকে গ্রিন-এ সুইচ করা হয়। কোনো সমস্যা হলে সাথে সাথে রোলব্যাক করে ব্লু-তে ট্রাফিক ঘুরিয়ে দেওয়া হয়। এতে ট্রাফিক লস জিরো থাকে, কিন্তু ক্লাউড কস্ট ডুপ্লিকেট বা দ্বিগুণ হয়।

#### ②. Canary Deployments (ক্যানারি ডিপ্লয়মেন্ট):
এটি অত্যন্ত সাশ্রয়ী ও সেফ। এখানে নতুন রিলিজের পর ট্রাফিকের একটি ক্ষুদ্র অংশ (যেমন: ৫%) নতুন ভার্সনে পাঠানো হয় এবং বাকি ৯৫% পুরানো ভার্সনে থাকে। সময়ের সাথে সাথে ৫% থেকে বাড়িয়ে ১০%, ৫০% হয়ে শেষে ১০০% করা হয়।

```
                  [API Gateway / Traffic Router]
                  / (90% Traffic)  \ (10% Traffic)
                 v                  v
    [Production V1 (Stable)]     [Canary V2 (Testing)]
```

* **Argo Rollouts ও Service Mesh:**
  ক্যানারি ট্রাফিক স্প্লিটিং করার জন্য কুবেরনেটিসে **Argo Rollouts** ব্যবহৃত হয়। এটি **Istio** বা **Linkerd** সার্ভিস মেশের এপিআই গেটওয়ে ব্যবহার করে রিয়েল-টাইমে সকেটের ৫% ট্রাফিক নতুন ক্যানারি পডে পাঠায় এবং রিয়েল-টাইমে প্রমিথিউস মেট্রিক্সে যদি এরর রেট বৃদ্ধি পায়, সে মানুষের হস্তক্ষেপ ছাড়াই অটোমেটিক ক্যানারি ডিলিট করে সেশন রোলব্যাক করে দেয়!

## ৩. Serverless Internals ও Edge Computing (Lambda & Cloudflare Workers)

সার্ভারলেস (Serverless) কম্পিউটিং হলো এমন একটি মডেল যেখানে ডেভেলপারকে কোনো ফিজিক্যাল বা ভার্চুয়াল সার্ভার ম্যানেজ করতে হয় না; ক্লাউড প্রোভাইডার ডাইনামিকালি ইউজারের কোড এক্সিকিউশনের জন্য কম্পিউট রিসোর্স এলোকেট করে।

### ক. AWS Lambda ও Firecracker MicroVM Internals
AWS Lambda-এর ব্যাকগ্রাউন্ডে সিকিউর আইসোলেশন ও সুপার-ফাস্ট বুটিং নিশ্চিত করতে ব্যবহৃত হয় কাস্টম ওএস প্রযুক্তি **Firecracker**:

* **Firecracker MicroVM:**
  এটি প্রথাগত হেভি ভার্চুয়াল মেশিন (যেমন: QEMU/KVM) এবং সাধারণ লাইট কন্টেইনারের মধ্যবর্তী একটি বৈপ্লবিক হাইব্রিড রূপ। এটি লিনাক্সের **Kernel-based Virtual Machine (KVM)** ব্যবহার করে মাত্র ৫ মিলি-সেকেন্ডে একটি অত্যন্ত সুরক্ষিত, মিনিমালিস্টিক মাইক্রো-ভিএম (MicroVM) ক্র্যাপ করে জেনারেট করতে পারে।
* **কোল্ড স্টার্ট (Cold Start) এবং অপ্টিমাইজেশন:**
  Lambda ফাংশন যখন অনেকক্ষণ অলস থাকে, AWS খরচ বাঁচাতে তার মাইক্রো-ভিএমটি কিল করে দেয়। অনেকদিন পর যখন প্রথম রিকোয়েস্ট আসে, তখন সম্পূর্ণ নতুন একটি মাইক্রো-ভিএম তৈরি করে, প্রজেক্টের নোড বা পাইথন রানটাইম লোড করে কোড রান করতে হয়। এই প্রথম রিকোয়েস্টের ল্যাটেন্সি ডিলে-কে **Cold Start** বলে।
  * **অপ্টিমাইজেশন (Provisioned Concurrency):** AWS ব্যাকগ্রাউন্ডে নির্দিষ্ট সংখ্যক মাইক্রো-ভিএম-এর মেমরি ইমেজ বা স্ন্যাপশট (MicroVM Snapshots) ডিস্কে ফ্রিজ করে রাখে। রিকোয়েস্ট আসা মাত্র মেমরি স্ন্যাপশট ডিরেক্ট র‍্যামে কপি করে বুটিং স্পিড ১০০০ গুণ বাড়িয়ে দেওয়া হয়।

### খ. Edge Computing: Cloudflare Workers ও V8 Isolates
প্রথাগত ক্লাউড ডাটা সেন্টার থাকে নির্দিষ্ট রিজিয়নে (যেমন: ওহাইও বা সিঙ্গাপুর)। কিন্তু ইউজার যদি বাংলাদেশ থেকে হিট করেন, ডেটা সিঙ্গাপুর ঘুরে আসতে ল্যাটেন্সি পেনাল্টি খাবে। **Edge Computing** ইউজারের কোড সরাসরি নিকটবর্তী সিডিএন এজ রাউটারে (CDN Edge Router) এক্সিকিউট করে।

#### V8 Isolates বনাম কন্টেইনার আর্কিটেকচারাল যুদ্ধ:
ক্লাউডফ্লেয়ার ওয়ার্কার্স (Cloudflare Workers) কোনো কন্টেইনার বা মাইক্রো-ভিএম টেকনোলজি ব্যবহার করে না। তারা সরাসরি গুগল ক্রোম ব্রাউজারের বিখ্যাত **Google V8 Javascript Engine** কার্নেল লেভেলে ব্যবহার করে।

| ফিচারের তুলনা | কন্টেইনার বা মাইক্রো-ভিএম (AWS Lambda) | V8 Isolates (Cloudflare Workers) |
| :--- | :--- | :--- |
| **কোল্ড স্টার্ট ল্যাটেন্সি** | ১০০ মিলি-সেকেন্ড থেকে ৩ সেকেন্ড | **০ থেকে ৫ মিলি-সেকেন্ড! (Zero Cold Start)** |
| **মেমরি ফুটপ্রিন্ট** | প্রতিটি কন্টেইনারে নূন্যতম ৩৫ মেগাবাইট | **প্রতিটি আইসোলেটে মাত্র ৩ মেগাবাইট!** |
| **আইসোলেশন বাউন্ডারি** | লিনাক্স নেমস্পেস ও সেককম্প ফিল্টার | গুগল ভি8 রানটাইম জাভাস্ক্রিপ্ট স্যান্ডবক্স |
| **স্কেলিং ক্যাপাসিটি** | মিডিয়াম (কন্টেইনার বুট টাইম সাপেক্ষ) | আল্ট্রা-ফাস্ট (একই ওএস প্রসেসে লাখ লাখ রান করা যায়) |

* **V8 Isolates ম্যাজিক:**
  একটি সিঙ্গেল ওএস প্রসেসের ভেতরে V8 ইঞ্জিন লাখ লাখ সম্পূর্ণ স্বাধীন জাভাস্ক্রিপ্ট মেমরি স্পেস তৈরি করতে পারে, যাদের একে অপরের ডেটা রিড করার কোনো সুযোগ নেই (Isolates)। ফলে কোনো বুটিং বা ওএস কার্নেল লেভেলের ওভারহেড ছাড়াই মিলি-সেকেন্ডের মধ্যে কোড রান করে রেসপন্স ব্যাক করা যায়।
